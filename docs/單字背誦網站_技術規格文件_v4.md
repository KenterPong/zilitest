# 單字背誦網站 技術規格文件(v4)

> 本文件聚焦技術實作細節(資料庫設計、業務邏輯、演算法規則),對應「單字背誦網站_營運文件.md」中定案的業務規則。
> 前端頁面與程式碼實作交由開發工具(如 Cursor)處理,本文件僅提供架構與邏輯依據。

## 版本更新說明(v4,取代 v3)

- **完全移除「難易度(difficulty)」欄位與相關功能**:單字不再需要手動標記難易度,`words` 表移除 `difficulty` 欄位。使用者只需要透過「單字本 + 標籤」自行篩選要背誦/測驗的範圍即可,不需要額外的難易度分級。
- 其餘邏輯與 v3 相同(重測答錯單字功能已移除、測驗出題為錯誤率加權隨機、單字本層級聚合統計已移除、500 字上限為帳號全站加總)。

## 一、帳號與訂閱狀態機

### 1. 狀態定義

| 狀態 | 說明 | 觸發者 |
|---|---|---|
| `trial` | 試用中,30 天,單字上限 500 | 系統(註冊時自動設定) |
| `active` | 付費中,正常扣款,無單字上限 | 系統(扣款成功) |
| `payment_failed` | 扣款失敗,進入 10 天寬限期,功能仍可正常使用 | 系統(扣款失敗) |
| `suspended` | 已暫停,全部功能鎖定(僅可匯出資料) | 系統(試用期滿未付費 / 寬限期滿未扣款成功) |
| `cancelled` | 已取消/刪除,資料清空 | 使用者主動操作,或系統於保留期滿後自動清除 |

### 2. 狀態轉換邏輯

```
trial ──(30天內完成付款)──> active
trial ──(30天到期未付款)──> suspended
trial ──(使用者主動刪除帳號)──> cancelled

active ──(每月扣款成功)──> active(更新 next_billing_at)
active ──(扣款失敗)──> payment_failed
active ──(使用者點「取消訂閱」,auto_renew=false)──> active(維持可用,不再續扣)
active(auto_renew=false) ──(到達 next_billing_at,略過扣款)──> suspended
active ──(使用者主動刪除帳號,不論 auto_renew 為何)──> cancelled

payment_failed ──(10天內補款成功)──> active
payment_failed ──(10天寬限期滿仍失敗)──> suspended
payment_failed ──(使用者主動刪除帳號)──> cancelled

suspended ──(使用者完成付款)──> active
suspended ──(使用者主動刪除帳號)──> cancelled
suspended ──(3個月保留期滿,資料清除排程執行)──> cancelled

cancelled ──(使用者以同一LINE帳號再次登入,視為全新帳號)──> trial
```

> 保留期滿觸發的清除視同帳號取消:資料清除完成後 `status` 一律轉為 `cancelled`(不維持在 `suspended`),避免系統中出現「資料已空但狀態仍是暫停」的曖昧狀態。
>
> `cancelled` 並非真正的終態:使用者的 `line_user_id` 資料列保留,若日後以同一 LINE 帳號再次登入,系統視為全新帳號,重新進入 `trial` 並給予完整 30 天試用期(見第二節 3.1)。
>
> **「取消訂閱」與「刪除帳號」是兩個獨立操作,定案如下:**
> - **取消訂閱**(僅限 `active`):使用者點擊後只設定 `auto_renew = false`,`status` 維持 `active`,功能持續可用至 `next_billing_at` 當天;排程於 `next_billing_at` 檢查到 `auto_renew = false` 時,略過扣款、直接將 `status` 轉為 `suspended`(不進入 `payment_failed`,因為這不是扣款失敗,而是使用者主動選擇不續訂)。使用者可在到期前隨時把 `auto_renew` 改回 `true` 取消這個決定,恢復正常續訂。
> - **刪除帳號**(所有狀態皆可執行,包含 `trial`/`active`/`payment_failed`/`suspended`):不論目前處於哪個狀態、剩餘多少天數或試用額度,一律**立即**清空資料並將 `status` 轉為 `cancelled`,**不退款、不等到期**,視同使用者主動放棄剩餘權益。`active` 使用者可以不先取消訂閱就直接刪除帳號,行為與上述「取消訂閱後續用到期」互斥且優先——一旦執行刪除,直接跳過續用期,立即生效。

### 4. 功能權限對照表

| 狀態 | 使用功能 | 新增單字 | 資料匯出 |
|---|---|---|---|
| trial | ✅ 全功能,500字上限 | ✅(未達上限) | ✅ |
| active | ✅ 全功能,無上限 | ✅ | ✅ |
| payment_failed | ✅ 全功能(寬限期內視同 active) | ✅ | ✅ |
| suspended | ❌ 全部功能鎖定 | ❌ | ✅(可下載後刪除帳號) |
| cancelled | — 帳號已刪除 | — | — |

### 5. 500 字上限檢查邏輯

> ⚠️ **重要:此上限是帳號層級的全站加總,不是單一單字本的上限。** 使用者可以有多個單字本,500 字上限是這些單字本裡的單字數量「全部加起來」的總數,不是每個單字本各自可以存到 500 字。UI 上不論在哪個頁面顯示這條上限(首頁、單字本內、匯入預覽),數字來源都必須是 `users.word_count`,不能用單一單字本自己的字數去比對 500。

- 檢查時機:**新增/匯入單字送出時**一次性檢查(不逐字檢查),用「目前 word_count + 本次新增數量」是否超過 500 判斷
- 若超過:整批擋下或提示使用者「已達上限,僅能匯入至上限為止」
- word_count 於批次操作完成後,重新計算並寫入(非即時逐筆更新)

### 6. 通知機制

- **登入時提醒**:一律顯示,不論有無 email。依 `status` 顯示對應 banner:
  - `trial` 快到期 → 顯示剩餘天數
  - `payment_failed` → 顯示寬限期剩餘天數
  - `suspended`(在 3 個月保留期內)→ 顯示資料保留剩餘天數提醒
- **Email 提醒**:僅 `email IS NOT NULL` 時寄送,寄送前比對 `last_reminder_sent_at` 避免當月重複寄送
- **帳號設定頁文案**:需明確告知「填寫 Email 才能收到到期/扣款提醒通知信,未填寫僅能透過登入畫面查看提醒」

### 7. 暫停帳號資料保留

- `suspended` 狀態資料保留 **3 個月**(`data_purge_scheduled_at` = suspended_at + 3 個月)
- 保留期間:每月檢查是否已寄送當月提醒(比對 `last_reminder_sent_at`),有 email 則寄信,登入時一律顯示 banner
- 3 個月到期後觸發清除資料排程(MVP 階段可先手動執行,不急著自動化),**清除完成後 `status` 轉為 `cancelled`**,與使用者主動刪除帳號視為同一終態

## 二、LINE Login 技術串接

> 沿用另一專案(AI 預約平台)已上線驗證過的架構,不使用 Supabase Auth,採手動 OAuth 流程。

### 1. LINE Developers Console 設定
- 建立 **LINE Login** channel(非 Messaging API channel)
- Callback URL:`https://你的網域/auth/callback`
- Scopes:僅勾選 `profile`、`openid`(**不需要 `email`**,email 由使用者自行於帳號設定頁填寫)
- 取得 `Channel ID`(LINE_CLIENT_ID)與 `Channel Secret`(LINE_CLIENT_SECRET)

### 2. 手動 OAuth 流程

```
1. 前端導向 LINE 授權頁(帶 client_id、redirect_uri、state)
2. LINE 導回 /auth/callback?code=xxx&state=xxx
3. 前端頁面驗證 state,呼叫後端 API
4. 後端 API:
   a. 用 code 換 LINE access token
   b. 用 access token 取得 LINE profile(userId、displayName、pictureUrl)
   c. UPSERT users 資料表(line_user_id 為唯一鍵)
   d. 寫入 httpOnly cookie(user_id)
   e. redirect 到主頁面
```

### 3. 首次登入建檔(範例邏輯)

```typescript
const { data: user } = await supabaseAdmin
  .from('users')
  .upsert(
    {
      line_user_id: lineProfile.userId,
      display_name: lineProfile.displayName,
      avatar_url: lineProfile.pictureUrl,
      // 僅在「新使用者」情況下需要設定以下三欄;若為既有使用者的重複登入,
      // upsert 不應覆蓋已存在的 status / trial_start_at / trial_end_at。
      // 實作上建議先查詢 line_user_id 是否已存在,不存在才帶入這三個欄位:
      status: 'trial',
      trial_start_at: new Date().toISOString(),
      trial_end_at: new Date(Date.now() + 30 * 24 * 60 * 60 * 1000).toISOString(),
    },
    { onConflict: 'line_user_id', ignoreDuplicates: false }
  )
  .select()
  .single()

response.cookies.set('user_id', user.id, {
  httpOnly: true,
  secure: true,
  sameSite: 'lax',
  maxAge: 60 * 60 * 24 * 30,
})
```

> ⚠️ 注意:`upsert` 若對既有使用者重複執行,不能每次都重設 `trial_start_at`/`status`,否則會讓老使用者的試用期被意外重新計算。實作時建議改用「先 `select` 判斷是否存在 → 不存在才 `insert` 帶入試用欄位 → 存在則只 `update` display_name/avatar_url」,而非單純依賴 `upsert` 一次帶入全部欄位。

### 3.1 `cancelled` 帳號再次登入的處理(定案)

- 資料清除完成、`status = cancelled` 後,`line_user_id` 這筆使用者資料列本身**不刪除**(僅清空其名下的 wordbooks/words/quiz 相關資料),保留作為登入識別之用
- 使用者日後用同一個 LINE 帳號再次登入時,**視為全新帳號**,重新給予 30 天試用期:
  - 判斷依據:查到 `line_user_id` 已存在但 `status = cancelled`
  - 處理方式:比照全新使用者,重設 `status = 'trial'`、`trial_start_at = now()`、`trial_end_at = now() + 30天`,並將 `word_count` 歸零
  - **完整重置欄位清單**(避免舊帳號殘留資料影響新一輪試用期或誤觸發扣款排程):`auto_renew = true`、`first_paid_at = NULL`、`next_billing_at = NULL`、`payment_failed_at = NULL`、`grace_period_end_at = NULL`、`suspended_at = NULL`、`data_purge_scheduled_at = NULL`、`last_reminder_sent_at = NULL`、`cancelled_at = NULL`
- 首次登入建檔邏輯(見上方 3.)需擴充判斷條件:`line_user_id` 不存在 **或** 已存在但 `status = cancelled`,兩種情況都要重新帶入試用三欄位;僅當 `line_user_id` 存在且 `status` 非 `cancelled` 時,才只更新 `display_name`/`avatar_url`

### 4. 授權規則

- 不使用 RLS,所有資料存取走「API + service role」
- `SUPABASE_SERVICE_ROLE_KEY` 僅能在伺服器端使用,不可用 `NEXT_PUBLIC_*` 前綴
- 所有讀寫都需在 API 層驗證 `cookie.user_id === row.user_id`
- `REVOKE ALL ON TABLE ... FROM anon, authenticated`,避免繞過 API 直接用 anon key 存取
- **`cancelled` 狀態的 session 攔截**:若使用者刪除帳號或資料被清除排程處理時,前端仍持有尚未過期的 `user_id` cookie,後續任何 API 請求須在驗證階段檢查對應 `users.status`;若查到 `status = 'cancelled'`,一律回傳 401 並清除該 cookie,前端導向登入頁。使用者重新以 LINE 登入時,會依 3.1 節邏輯自動視為全新帳號、重置為 `trial`,不需要額外的「復活」流程

### 5. 設計原則

- 標籤:使用者自訂、可多選、可跨單字本,用於複習/測驗時的複合篩選
- 篩選邏輯:**多選單字本時,單字本與單字本之間為聯集(OR)**——因每個單字只屬於一個 `wordbook_id`,若單字本之間也用 AND,篩選結果必為空集合。「單字本(聯集)」與「標籤」之間為交集(AND);**多選標籤時,標籤與標籤之間也是交集(AND)**——即同時選「動詞」+「易錯字」,篩選結果須同時具備這兩個標籤的單字,而非只符合其中之一。整體篩選邏輯為:`(單字本A 或 單字本B 或...) 且 (標籤A 且 標籤B 且...)`。前端與後端(SQL 查詢)皆須依此實作,避免出現邏輯誤解
- **空篩選條件的定義**:若 `filter_tag_ids` 為空陣列,略過標籤條件,僅依單字本篩選(不代表「篩選出沒有任何標籤的字」)。若 `filter_wordbook_ids` 為空陣列,視為尚未選擇範圍,**不允許開始測驗/背誦**,前端應擋下並提示使用者至少選擇一本單字本

## 四、Excel 匯入規則

- 匯入欄位僅 **3 欄**:單字(term)、答案(answer)、描述(description)
- 標籤**不開放於匯入時設定**,一律匯入後由使用者手動於單字下設定,避免匯入格式對不上標籤造成錯誤
- 匯入流程:
  1. 使用者上傳 Excel/CSV
  2. 前端使用 SheetJS(xlsx.js)解析
  3. 顯示預覽畫面供使用者確認
  4. 送出前檢查:目前 `word_count`(⚠️ 帳號全站加總,非單一單字本)+ 本次匯入筆數 是否超過 500(試用期)上限
  5. 超過上限 → 提示訊息,可選擇僅匯入至上限,或先升級付費版
  6. 確認送出後寫入 `words` 表,並更新 `users.word_count`

## 五、測驗模組

### 1. `word_stats`(單字統計)

| 欄位 | 型別 | 說明 |
|---|---|---|
| `word_id` | UUID FK | 一對一對應 words |
| `user_id` | UUID FK | 冗餘存,方便查詢 |
| `attempt_count` | INTEGER DEFAULT 0 | 已測驗次數 |
| `correct_count` | INTEGER DEFAULT 0 | 答對次數 |
| `last_tested_at` | TIMESTAMPTZ (nullable) | 最後一次測驗時間 |

> `accuracy`(正確率)不存欄位,即時計算 `correct_count / attempt_count`,避免額外同步成本。此正確率**僅為單字層級**的統計,不對單字本做聚合計算,首頁與單字本列表也不顯示單字本層級的正確率或最近測驗時間彙總。

### 3. `quiz_answers`(每題作答紀錄)

| 欄位 | 型別 | 說明 |
|---|---|---|

| `session_id` | UUID FK | |
| `word_id` | UUID FK | |
| `is_correct` | BOOLEAN | 是否答對 |

> 不儲存使用者原始輸入內容(`user_input`),測驗結束當下即時運算紅字標記後即捨棄,不留底。

### 4. 出題邏輯:錯誤率加權隨機(v3 修正)

> 測驗的單字本、標籤篩選**完全由使用者自行選擇**,系統不做「自動重測答錯單字」這類自動篩選功能。加權隨機只發生在「同一批篩選結果中,決定哪些字被抽到、抽到機率各是多少」這個層面。

1. 篩選出符合條件(單字本 + 標籤,皆由使用者手動選擇)的單字池
2. 計算池中每個單字的**抽選權重**:
   - 從未測驗過的單字(`attempt_count = 0`):給予最高權重(視同錯誤率 100%)
   - 已測驗過的單字:`weight = 1 + (1 - accuracy) × K`,`K` 為加權係數(建議先設為 4,之後可依實測調整),即錯誤率越高權重越高,錯誤率 0% 時權重最低(=1)
3. 使用「加權隨機抽樣、不放回」的方式,依權重抽出使用者設定的題數(推薦實作方式:Efraimidis-Spirakis 演算法——每個單字產生亂數 key = `random()^(1/weight)`,依 key 由大到小排序,取前 N 個)
4. 若使用者設定題數超過池子總數 → 全部單字入選,並提示實際題數較少
5. 加權隨機的設計意圖:讓錯誤率高的單字更常出現,藉此加強不熟單字的複習,但不是每次都固定只考錯的字,仍保留隨機性與正確率高的字偶爾出現的機會

### 5. 選擇題/是非題出題防呆

- 選擇題:單字池需至少 **4 個單字**(1 正確 + 3 干擾項),不足需提示使用者擴大篩選範圍或改選其他題型
- 是非題:單字池需至少 **2 個單字**(否則無法出「錯誤配對」)
- 檢查時機:使用者設定測驗條件、送出開始測驗前

### 6. 是非題邏輯

- 出題時先隨機決定「本題問正確配對」或「本題問錯誤配對」(各半機率)
- 若為錯誤配對 → 該單字的 `answer` 換成同一單字池中隨機抽取的另一個單字的 `answer`
- 使用者判斷「單字與中文配對是否正確」

### 7. 選擇題干擾項邏輯

- 正確答案:該單字本身的 `answer`
- 干擾項:**從同一篩選後的單字池中完全隨機抽取**其他單字的 `answer`(MVP 不做相似度比對,先求功能可用)

### 8. 輸入題判定邏輯(純程式碼,不呼叫 API)

- **判定標準:嚴格完全一致**(忽略大小寫、去除頭尾空白),逐字元有任何差異(多字/少字/拼錯/順序錯)一律判定 `is_correct = false`
- **紅字標記邏輯**(僅測驗結束後、結果頁一次顯示,不即時顯示,不存資料庫):
  - 使用 Levenshtein 對齊演算法找出「插入/刪除/替換」的具體位置
  - 替換錯誤(拼錯字母)→ 標記該字母為紅色
  - 少字母 → 標記缺少位置(底線或插入符號)
  - 多字母 → 標記多餘字母(紅色刪除線)
  - 若差異過大(完全不相關的答案)→ 不強行標紅,直接顯示正確答案

## 六、卡牌背誦模式

### 1. `card_familiarity`(卡牌熟悉度自評)

| 欄位 | 型別 | 說明 |
|---|---|---|
| `word_id` | UUID FK | |
| `user_id` | UUID FK | 冗餘存,方便查詢 |
| `familiarity` | TEXT | `unknown`(不認得)/ `known`(認得) |
| `updated_at` | TIMESTAMPTZ | 最後標記時間 |

> 與 `word_stats`(測驗統計)完全獨立,互不影響彼此邏輯運算。卡牌模式的加權是二元的(認得/不認得各一個固定倍率),與測驗模式「依連續型錯誤率計算權重」是兩套獨立邏輯,不要合併實作。

### 2. 篩選邏輯
- 與測驗模式共用同一套「單字本 + 標籤」篩選 UI(皆由使用者手動選擇)
- 篩選完成後可直接串接進入測驗(背完可馬上測)

### 3. 排序邏輯:加權隨機
- `familiarity = unknown` 或**尚未標記過**的單字 → 權重較高(如 3 倍機率),兩者權重相同
- `familiarity = known` → 權重較低(如 1 倍機率),但不完全排除,仍有機會出現
- 使用者可標記「認得 / 不認得」,寫入 `card_familiarity`
